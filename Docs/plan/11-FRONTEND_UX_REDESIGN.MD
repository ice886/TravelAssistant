# FRONTEND UX REDESIGN

> 日期：2026-07-12  
> 类型：Plan  
> 状态：已确认，进入开发

## 1. 目标

重新设计 TravelAssistant 前端工作台，在不改变后端 API、数据库 Schema 和 Agent 编排逻辑的前提下，提升从创建旅行、执行研究、查看证据到生成与编辑行程的整体清晰度、可信度和使用体验。

设计方向为可信、清晰、温暖的旅行规划工作台。桌面端保持适中的信息密度，平板和移动端采用自然重排的分阶段纵向流程。

## 2. 用户与成功标准

主要用户是在有限时间内整理旅行需求、等待多来源研究并形成可执行行程的旅行者。

核心成功标准：

- 用户能够明确理解当前处于创建、研究还是行程编辑阶段。
- 主要操作与下一步行动在各阶段保持清晰可见。
- Research 的运行、缓存命中、降级、失败和完成状态具有可理解的反馈。
- 来源证据与行程结果之间的关系清楚，并保留系统可信度。
- 页面在桌面、平板和移动端可用，键盘操作与颜色对比达到 WCAG 2.2 AA 基础要求。

## 3. Skill 方法

- `design-brief`：约束问题、用户、范围和可衡量的成功标准。
- `interfaces-that-feel`：围绕规划、等待、成功和错误恢复设计文案、节奏与交互反馈。
- `responsive-design`：使用内容驱动、移动端自然重排的响应式策略。
- `accessibility-audit`：按 WCAG 2.2 POUR 原则检查语义、键盘、焦点、对比度与辅助技术兼容性。
- `design-critique`：从用户目标出发识别单页滚动、操作竞争和状态不可见问题。
- `information-architecture`：按“描述旅程、收集证据、编辑行程”建立浅层任务结构。
- `navigation-patterns`：使用阶段导航和明确的 active/disabled 状态支持跨阶段定位。
- `visual-hierarchy`：每个阶段只保留一个主要行动，降低同时展示内容的竞争。

## 4. 执行步骤

1. 审计现有 `App`、旅行表单、Research 进度、来源面板和行程编辑器，确认组件边界、状态模型、样式体系与主要体验问题。
2. 建立统一设计语言，定义色彩、字体、间距、圆角、阴影、布局和状态反馈 Token。
3. 重构工作台信息架构与页面层级，清晰表达“创建旅行 → 收集证据 → 生成/编辑行程”的主流程。
4. 重设计核心组件，补齐 loading、empty、error、degraded、success、disabled 等状态与对应微文案。
5. 完成桌面、平板、移动端响应式布局，并完善语义 HTML、键盘焦点、触摸目标、对比度和 reduced-motion。
6. 运行 lint、typecheck、test 与 build，人工复核可读性和交互状态，并更新相关项目文档与 daily log。

## 9. 单页阶段工作台优化（追加需求）

> 状态：已确认，进入开发

当前单页同时展开旅行表单、Research、来源和行程编辑，导致用户必须连续滚动才能完成任务。本次追加优化采用“单页、多阶段视图”，不引入路由和新的状态管理：

1. 顶部阶段导航保持可见，并显示当前阶段、已完成阶段和不可用阶段。
2. 当前阶段只渲染对应工作区：描述旅程显示表单，收集证据显示 Research 与来源，编辑行程显示行程编辑器。
3. 创建旅行后自动进入收集证据；研究完成后自动进入编辑行程；用户可返回已完成阶段查看内容。
4. 每个阶段提供明确的上一步/下一步操作，前置条件不满足时禁用并解释原因。
5. 桌面端使用紧凑的顶部阶段导航，移动端使用可横向滚动但保持可见的导航；所有目标保持至少 44px。

## 10. 第三阶段旅行计划紧凑化（追加需求）

> 状态：已确认，进入开发

1. 第三阶段前端移除可编辑行程能力，不再展示标题、摘要、时间、活动说明输入框和保存按钮。
2. 保留行程生成与最新行程读取，后端生成/保存 API 契约暂不删除。
3. 将行程展示改为紧凑的只读活动列表：左侧类型图标，中间标题/必去标签/描述，右侧时间/费用。
4. 使用低干扰卡片、紧凑间距和单一视觉重点提升扫描效率；移动端保持费用与时间可见且不产生横向溢出。

## 11. Hero 文案与旅行计划图标（追加需求）

> 状态：已完成

1. Hero 标题改为“去有风的地方”，同步缩短 Hero 垂直空间。
2. Hero 描述改为强调输入目的地和生成清晰旅行计划，移除“可编辑行程草稿”表述。

4. 旅行计划活动卡片使用 24px 线性 SVG 图标，按交通、景点/艺术、自然、公园、地标和餐饮内容匹配颜色与图形。

## 6. 约束与风险

- 不引入新的组件库、状态管理或数据请求依赖，优先复用现有 React/Vite 技术栈。
- 不修改后端接口契约；现有异步轮询、降级和不可变行程版本语义必须保留。
- 地图、PDF 导出、Zustand 和 React Query 仍属于 v2 后续目标，不在本次范围内。
- 重设计需要覆盖复杂异步状态，必须避免只优化正常流程而遗漏失败、部分数据和恢复体验。
